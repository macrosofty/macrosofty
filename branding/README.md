# Branding — where every visible bit lives

Single source of truth → derived files. This file is the lookup table for
"I want to change X, where do I edit?" Edit the source, run the regen
script (when there is one), commit the diff.

## Quick start

```bash
# Change a brand string (URL, tagline, name) — no script needed
$EDITOR config/identity.env

# Change the doorway icon, wallpaper, or Anaconda banner
$EDITOR branding/logo-master.svg                       # or
$EDITOR branding/themes/saffron/source/wallpaper.svg   # or
$EDITOR branding/tjopper/anaconda-header.svg
scripts/regenerate-branding.sh
```

## Brand strings

Single file: [`config/identity.env`](../config/identity.env). Every
string the user reads (distro name, taglines, URLs, default hostname,
edition list, AppHelperThingy pin) lives there.

Picked up automatically by:

| Script | When it runs | What it brands |
|---|---|---|
| `scripts/generate-os-release.sh` | OCI build (per edition) | `/etc/os-release`, `/etc/macrosofty-release` |
| `scripts/scrub-upstream-branding.sh` | OCI build (per edition) | Inherited Aurora/Bazzite `.desktop`, motd, profile.d greetings |
| `scripts/rebrand-netinstall-iso.sh` | Netinstall ISO build | Anaconda chrome `/.buildstamp`, os-release, system-release, redhat-release |

If you change `BRAND_NAME` to "Bigsoftie" in identity.env and rebuild,
all four editions and the netinstall ISO will say "Bigsoftie" without
any other code change. That's the contract.

## Anaconda installer chrome — gotchas worth knowing

The netinstall installer chrome ("FEDORA 43 INSTALLATION" → "MACROSOFTY
PADKOS 43 INSTALLATION") is rebranded at ISO build time by
[`scripts/rebrand-netinstall-iso.sh`](../scripts/rebrand-netinstall-iso.sh).

The visible chrome has **four** independent sources, and missing any
one of them leaks Fedora through:

| Surface | Where the bytes come from | Where to change it |
|---|---|---|
| Title bar text — "MACROSOFTY 43 INSTALLATION" | `/.buildstamp` `[Main] Product` + `Version`, **not** os-release | rebrand script rewrites Product/BugURL/Variant in place using `config/identity.env` |
| Sidebar background image | `/usr/share/anaconda/pixmaps/sidebar-bg.png` | overlaid from `system_files/shared/usr/share/anaconda/pixmaps/sidebar-bg.png` |
| Sidebar wordmark / icon | `/usr/share/anaconda/pixmaps/sidebar-logo.png` (the upstream "fedora" wordmark — easy to miss) | overlaid from `system_files/shared/usr/share/anaconda/pixmaps/sidebar-logo.png` |
| Top header banner | `/usr/share/anaconda/pixmaps/anaconda_header.png` | overlaid from `system_files/shared/usr/share/anaconda/pixmaps/anaconda_header.png` |

The product config at
`system_files/shared/usr/share/anaconda/product.d/macrosofty.conf`
declares the product to Anaconda but does **not** drive the title text —
that is `pyanaconda/core/product.py:get_product_values()` reading
`/.buildstamp`.

See [`anaconda/README.md`](anaconda/README.md) for the design rationale
behind the surgical-overlay approach (vs. custom Lorax templates or a
`macrosofty-logos` RPM).

## Theme pack — applied at OCI build time

`macrosofty-theme apply saffron` runs in every edition's `build.sh` and
applies the theme pack at
`system_files/shared/usr/share/macrosofty/themes/saffron/`. Pack format
documented in [`pack.json`](../system_files/shared/usr/share/macrosofty/themes/saffron/pack.json)
and the `macrosofty-theme` script.

| Component | File in pack | Where it lands on disk |
|---|---|---|
| Kickoff (start-menu) icon | `icons/start-here-kde.svg` | `/usr/share/icons/hicolor/scalable/places/start-here-*.svg` and overwrites Aurora's `distributor-logo*.svg` |
| Wallpaper | `wallpaper-1920x1080.png` | `/usr/share/wallpapers/Macrosofty/contents/images/1920x1080.png` |
| MOTD | `motd` | `/etc/motd` |
| GRUB distributor | string in `pack.json` | `/etc/default/grub.d/99-macrosofty.cfg` |
| Plymouth theme | `plymouth/` | `/usr/share/plymouth/themes/macrosofty/`, plus alternatives + plymouthd.conf wiring |
| SDDM background | `login-bg.png` | `/usr/share/sddm/themes/breeze/Background.png` |
| Look-and-Feel | `lookandfeel/` (placeholder) | `/usr/share/plasma/look-and-feel/com.macrosofty.desktop/` |

To add a second theme pack: copy
`system_files/shared/usr/share/macrosofty/themes/saffron/` to a sibling
directory, edit `pack.json`, drop your assets in. Users apply with
`sudo macrosofty-theme apply <name>`.

The `tjopper` pack at
`system_files/shared/usr/share/macrosofty/themes/tjopper/` is a worked
example of a *partial* pack: it ships only `wallpaper` + `loginBackground`
+ `sddmBackground`, so applying it changes the desktop and SDDM
backgrounds while leaving kickoff icon, Plymouth, motd, and GRUB
untouched (they keep whatever the previously-applied pack had set).
`macrosofty-theme apply tjopper` is a no-op for unset components — the
contract is "missing → skip", never "missing → reset".

## Hand-tuned, not regenerated

The script leaves these alone — edit the source directly:

- **Kickoff-icon SVGs** at
  `system_files/shared/usr/share/macrosofty/themes/saffron/icons/start-here-*.svg`.
  `start-here-kde.svg` and `start-here.svg` are nearly the same as
  `logo-master.svg` but with KDE-specific desc text.
  `start-here-symbolic.svg` is a 16×16 monochrome variant for status-bar
  contexts — completely different from the master logo.
- **Plymouth theme `.script`/`.plymouth`** at
  `system_files/shared/usr/share/macrosofty/themes/saffron/plymouth/`.
  These are Plymouth DSL files, not assets. Edit if you want a
  different boot animation.
- **Tjopper anaconda-header.svg** is the source for the Anaconda
  header banner above; it's a hand-composed layout (Tjopper on the
  left, wordmark on the right) — not derived from logo-master.svg.
- **`branding/iterations/`** — design history of logo proposals from
  the early sessions. Not consumed by the build.

## Adding a new branded surface

1. Decide which source it derives from (`logo-master.svg`,
   `wallpaper.svg`, `anaconda-header.svg`, or a new source).
2. Add the render step to `scripts/regenerate-branding.sh` in the
   appropriate section.
3. Add a row to the "source → derived map" table above.
4. Run the script and commit the new asset under `system_files/shared/`.

If the new asset is a *runtime* string-derived file (like os-release
or the netinstall buildstamp), wire it into `generate-os-release.sh`,
`scrub-upstream-branding.sh`, or `rebrand-netinstall-iso.sh` instead —
those run at build time and read `config/identity.env`.
