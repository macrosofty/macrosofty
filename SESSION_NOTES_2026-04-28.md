### Token plumbing (operational, persists across sessions via memory)
- Discovered each session was rediscovering the GitHub token and re-fumbling Cloudflare/GHCR credentials. Built a single load helper: `~/.config/macrosofty/load-tokens.sh` exports `GITHUB_TOKEN`, `GH_TOKEN`, `GHCR_TOKEN`, `CF_API_TOKEN`, `CF_ACCOUNT_ID` from three known files.
- Reasons for separation of concerns: the GitHub commit token (`~/.git-credentials`) lacks package scopes by design — having a dedicated GHCR PAT (`~/.config/macrosofty/ghcr-token`) keeps least-privilege.
- Restored persistent podman GHCR auth at `~/.config/containers/auth.json` (the previous one was in `$XDG_RUNTIME_DIR`/tmpfs and got wiped by reboot). Added `REGISTRY_AUTH_FILE` export to `~/.bashrc` so podman/skopeo find it without flags.
- Verified end-to-end: `skopeo inspect docker://ghcr.io/macrosofty/hearty:latest` returns the manifest.

## Technical Decisions
- **One GitHub token covers commit + REST API**, separate token for GHCR. Two PATs, not three. Justification: the commit/API token doesn't need package scopes (least-privilege); the GHCR token doesn't need repo scope. Either can rotate without breaking the other workflow.
- **Podman auth must live at `~/.config/containers/auth.json`, never tmpfs.** The default `$XDG_RUNTIME_DIR/containers/auth.json` is wiped on reboot — a real footgun. Documented the recovery procedure in memory so it's a 30-second rebuild next time, not a 30-minute rediscovery.
- **CI flake fix is a retry, not a fix.** registry.fedoraproject.org's reliability is upstream — we can't fix it, but we can absorb 5 minutes of intermittent loss before failing.
- **"Netinstall" is jargon, dropped from website copy.** The Windows-tired user doesn't know the word; "Online installer" + size hint is plain language. Door-philosophy: explain it like at a braai.
